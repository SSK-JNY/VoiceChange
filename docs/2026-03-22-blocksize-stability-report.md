# ブロックサイズ安定化レポート（2026-03-22）

## 概要
今回の検証では、リアルタイム推論の安定性改善を目的にブロックサイズを段階的に調整した。
結論として、現状環境では 12288 と 16384 の設定で安定動作を確認した。

## 背景
- サンプルレート: 44100 Hz
- 既存設定での症状:
  - 推論処理時間がチャンク予算を超過しやすい
  - AudioStats にて RVC 処理がボトルネックとして継続的に観測される
- 代表ログ:
  - Total avg が約 236 から 240 ms
  - expect が約 185.8 ms
  - RVC が約 235 から 239 ms（主要ボトルネック）

## 実施内容
1. 推論パラメータ調整の試行
- f0_method を dio にすると環境依存でエラー再発が確認されたため、rmvpe を維持
- 軽量化パラメータ（index_rate, protect, filter_radius, rms_mix_rate）も試行したが、今回条件では改善せず

2. ブロックサイズ調整
- UI 側の候補値に 12288 と 16384 を追加
- 実運用にて 12288/16384 の安定動作を確認

## 結果
- 安定性:
  - 12288: 安定
  - 16384: 安定
- これまでの課題だったコールバック過負荷の発生頻度は低減傾向
- Phase 6 検証は通過状態を維持

## 技術的解釈
- 本件の主要律速は I/O ではなく RVC 推論本体
- ブロックサイズ拡大により、1チャンクあたりの処理予算が増え、推論完了までの余裕が確保された

## トレードオフ（重要）
ブロックサイズを上げることで安定性は向上するが、以下のデメリットがある。

- 体感遅延の増加
  - 8192: 約 186 ms
  - 12288: 約 279 ms
  - 16384: 約 372 ms
- 話し始め、停止、パラメータ変更反映の追従が遅くなる
- ドロップ発生時の 1 回あたり影響時間が長くなる

## 推奨運用
- 会話用途で遅延も重視する場合: 12288
- 安定性最優先で遅延許容できる場合: 16384
- f0_method は現時点では rmvpe 推奨（dio は環境依存エラーの再発履歴あり）

## 次アクション案
1. 12288 を基準値として常用し、長時間運用ログを収集
2. 必要時のみ 16384 へ切替える運用ルール化
3. 将来的には、ウォームアップ導入や高速モード比率調整で遅延と安定性のバランスを再最適化
